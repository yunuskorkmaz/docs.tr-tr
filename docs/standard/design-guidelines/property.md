# <a name="property-design"></a><span data-ttu-id="1553f-102">Özellik Tasarımı</span><span class="sxs-lookup"><span data-stu-id="1553f-102">Property Design</span></span>
<span data-ttu-id="1553f-103">Özellikler teknik açıdan yöntemlere çok benzer olsa da, bunlar kullanım senaryolarında oldukça farklıdır.</span><span class="sxs-lookup"><span data-stu-id="1553f-103">Although properties are technically very similar to methods, they are quite different in terms of their usage scenarios.</span></span> <span data-ttu-id="1553f-104">Akıllı alanlar olarak görülenmelidir.</span><span class="sxs-lookup"><span data-stu-id="1553f-104">They should be seen as smart fields.</span></span> <span data-ttu-id="1553f-105">Alanların çağırma söz dizimi ve yöntemlerin esnekliği vardır.</span><span class="sxs-lookup"><span data-stu-id="1553f-105">They have the calling syntax of fields, and the flexibility of methods.</span></span>

 <span data-ttu-id="1553f-107">Özelliğin türü kesilebilir bir başvuru türü ise, özellik salt al olsa bile Özellik değerinin değiştirilebileceğini aklınızda bulundurun.</span><span class="sxs-lookup"><span data-stu-id="1553f-107">Keep in mind that if the type of the property is a mutable reference type, the property value can be changed even if the property is get-only.</span></span>

 <span data-ttu-id="1553f-109">Örneğin, özellikleri ortak bir ayarlayıcı ve korumalı bir alıcı ile birlikte kullanmayın.</span><span class="sxs-lookup"><span data-stu-id="1553f-109">For example, do not use properties with a public setter and a protected getter.</span></span>

 <span data-ttu-id="1553f-110">Özellik alıcısı sağlanmadıysa, işlevi bir yöntem olarak uygulayın.</span><span class="sxs-lookup"><span data-stu-id="1553f-110">If the property getter cannot be provided, implement the functionality as a method instead.</span></span> <span data-ttu-id="1553f-111">Yöntem adını ile başlatmayı `Set` ve özelliği adlandırdıklarınız ile birlikte izlemenizi düşünün.</span><span class="sxs-lookup"><span data-stu-id="1553f-111">Consider starting the method name with `Set` and follow with what you would have named the property.</span></span> <span data-ttu-id="1553f-112">Örneğin, <xref:System.AppDomain> `SetCachePath` yalnızca set-Only özelliği olmak yerine çağrılan bir yöntemi vardır `CachePath` .</span><span class="sxs-lookup"><span data-stu-id="1553f-112">For example, <xref:System.AppDomain> has a method called `SetCachePath` instead of having a set-only property called `CachePath`.</span></span>

 <span data-ttu-id="1553f-113">✔️ Tüm özellikler için, varsayılan değerlerin bir güvenlik deliğe veya daha fazla verimsiz bir koda neden olmamasını sağlamak için, tüm özellikler için hazır varsayılan değerler sağlar.</span><span class="sxs-lookup"><span data-stu-id="1553f-113">✔️ DO provide sensible default values for all properties, ensuring that the defaults do not result in a security hole or terribly inefficient code.</span></span>

 <span data-ttu-id="1553f-114">✔️, nesnenin geçici olarak geçersiz bir durumuna neden olsa da, özelliklerin herhangi bir sırada ayarlanmasına izin verir.</span><span class="sxs-lookup"><span data-stu-id="1553f-114">✔️ DO allow properties to be set in any order even if this results in a temporary invalid state of the object.</span></span>

 <span data-ttu-id="1553f-115">İki veya daha fazla özelliğin, bir özelliğin bazı değerlerinin aynı nesne üzerindeki diğer özelliklerin değerleri verildiğinde geçersiz olabileceği bir nokta ile ilişkili olması yaygındır.</span><span class="sxs-lookup"><span data-stu-id="1553f-115">It is common for two or more properties to be interrelated to a point where some values of one property might be invalid given the values of other properties on the same object.</span></span> <span data-ttu-id="1553f-116">Bu gibi durumlarda, ilişkili özellikler gerçekten nesne tarafından birlikte kullanıldığından, geçersiz durumdan kaynaklanan özel durumlar ertelenmelidir.</span><span class="sxs-lookup"><span data-stu-id="1553f-116">In such cases, exceptions resulting from the invalid state should be postponed until the interrelated properties are actually used together by the object.</span></span>

 <span data-ttu-id="1553f-117">bir özellik ayarlayıcısı özel durum oluşturursa, önceki değeri koruma ✔️.</span><span class="sxs-lookup"><span data-stu-id="1553f-117">✔️ DO preserve the previous value if a property setter throws an exception.</span></span>

 <span data-ttu-id="1553f-118">❌Getters Özellikler 'den özel durumlar oluşturmaktan kaçının.</span><span class="sxs-lookup"><span data-stu-id="1553f-118">❌ AVOID throwing exceptions from property getters.</span></span>

 <span data-ttu-id="1553f-119">Özellik alıcıları basit işlemler olmalıdır ve herhangi bir önkoşullara sahip olmamalıdır.</span><span class="sxs-lookup"><span data-stu-id="1553f-119">Property getters should be simple operations and should not have any preconditions.</span></span> <span data-ttu-id="1553f-120">Alıcı bir özel durum oluştur, büyük olasılıkla bir yöntem olarak yeniden tasarlanması gerekir.</span><span class="sxs-lookup"><span data-stu-id="1553f-120">If a getter can throw an exception, it should probably be redesigned to be a method.</span></span> <span data-ttu-id="1553f-121">Bu kuralın, bağımsız değişkenlerin doğrulanması sonucunda özel durumlar beklediğimiz Dizin oluşturucular için uygulanmadığından emin olun.</span><span class="sxs-lookup"><span data-stu-id="1553f-121">Notice that this rule does not apply to indexers, where we do expect exceptions as a result of validating the arguments.</span></span>

### <a name="indexed-property-design"></a><span data-ttu-id="1553f-122">Dizinli özellik tasarımı</span><span class="sxs-lookup"><span data-stu-id="1553f-122">Indexed Property Design</span></span>
 <span data-ttu-id="1553f-123">Dizinli bir özellik, parametrelere sahip olabilecek ve dizi dizine benzer özel söz dizimi ile çağrılabilen özel bir özelliktir.</span><span class="sxs-lookup"><span data-stu-id="1553f-123">An indexed property is a special property that can have parameters and can be called with special syntax similar to array indexing.</span></span>

 <span data-ttu-id="1553f-130">Tasarım birden çok parametre gerektiriyorsa, özelliğin bir mantıksal koleksiyona bir erişimciyi temsil edip etmediğini yeniden düşünün.</span><span class="sxs-lookup"><span data-stu-id="1553f-130">If the design requires multiple parameters, reconsider whether the property really represents an accessor to a logical collection.</span></span> <span data-ttu-id="1553f-131">Yoksa, bunun yerine yöntemleri kullanın.</span><span class="sxs-lookup"><span data-stu-id="1553f-131">If it does not, use methods instead.</span></span> <span data-ttu-id="1553f-132">Veya ile yöntem adını başlatmayı düşünün `Get` `Set` .</span><span class="sxs-lookup"><span data-stu-id="1553f-132">Consider starting the method name with `Get` or `Set`.</span></span>

 <span data-ttu-id="1553f-133">❌,,, <xref:System.Int32?displayProperty=nameWithType> <xref:System.Int64?displayProperty=nameWithType> <xref:System.String?displayProperty=nameWithType> Veya enum dışında parametre türlerine sahip dizin oluşturuculardan kaçının <xref:System.Object?displayProperty=nameWithType> .</span><span class="sxs-lookup"><span data-stu-id="1553f-133">❌ AVOID indexers with parameter types other than <xref:System.Int32?displayProperty=nameWithType>, <xref:System.Int64?displayProperty=nameWithType>, <xref:System.String?displayProperty=nameWithType>, <xref:System.Object?displayProperty=nameWithType>, or an enum.</span></span>

 <span data-ttu-id="1553f-140">❌Hem bir Dizin Oluşturucu hem de anlamsal olarak eşdeğer Yöntemler sağlamaın.</span><span class="sxs-lookup"><span data-stu-id="1553f-140">❌ DO NOT provide both an indexer and methods that are semantically equivalent.</span></span>

### <a name="property-change-notification-events"></a><span data-ttu-id="1553f-145">Özellik değişiklik bildirimi olayları</span><span class="sxs-lookup"><span data-stu-id="1553f-145">Property Change Notification Events</span></span>
 <span data-ttu-id="1553f-146">Bazen bir özellik değerindeki değişiklikleri kullanıcıya bildiren bir olay sağlamak yararlı olur.</span><span class="sxs-lookup"><span data-stu-id="1553f-146">Sometimes it is useful to provide an event notifying the user of changes in a property value.</span></span> <span data-ttu-id="1553f-147">Örneğin, `System.Windows.Forms.Control` `TextChanged` özelliğinin değeri değiştirildikten sonra bir olay oluşturur `Text` .</span><span class="sxs-lookup"><span data-stu-id="1553f-147">For example, `System.Windows.Forms.Control` raises a `TextChanged` event after the value of its `Text` property has changed.</span></span>

 <span data-ttu-id="1553f-148">✔️, üst düzey API 'lerde (genellikle Tasarımcı bileşenleri) özellik değerleri değiştirildiğinde değişiklik bildirimi olaylarını yükseltmeyi göz önünde bulundurun.</span><span class="sxs-lookup"><span data-stu-id="1553f-148">✔️ CONSIDER raising change notification events when property values in high-level APIs (usually designer components) are modified.</span></span>

 <span data-ttu-id="1553f-149">Bir kullanıcının bir nesnenin özelliğinin ne zaman değiştiğini bilmesini sağlamak için iyi bir senaryo varsa, nesne özellik için bir değişiklik bildirimi olayı tetiklemelidir.</span><span class="sxs-lookup"><span data-stu-id="1553f-149">If there is a good scenario for a user to know when a property of an object is changing, the object should raise a change notification event for the property.</span></span>

 <span data-ttu-id="1553f-150">Bununla birlikte, temel türler veya Koleksiyonlar gibi alt düzey API 'Ler için bu tür olayları yükseltme yüküyle ilgili ek yükün olması düşüktür.</span><span class="sxs-lookup"><span data-stu-id="1553f-150">However, it is unlikely to be worth the overhead to raise such events for low-level APIs such as base types or collections.</span></span> <span data-ttu-id="1553f-151">Örneğin, <xref:System.Collections.Generic.List%601> listeye yeni bir öğe eklendiğinde ve özelliği değiştiğinde bu tür olayları oluşturmaz `Count` .</span><span class="sxs-lookup"><span data-stu-id="1553f-151">For example, <xref:System.Collections.Generic.List%601> would not raise such events when a new item is added to the list and the `Count` property changes.</span></span>

 <span data-ttu-id="1553f-152">✔️, bir özelliğin değeri dış güçlerle değiştirildiğinde değişiklik bildirimi olaylarını yükseltmeyi göz önünde bulundurun.</span><span class="sxs-lookup"><span data-stu-id="1553f-152">✔️ CONSIDER raising change notification events when the value of a property changes via external forces.</span></span>

 <span data-ttu-id="1553f-153">Bir özellik değeri, bazı dış zorlanarak (nesne üzerinde Yöntemler çağırma dışında bir şekilde) değişirse, olay oluşturma, geliştiricinin değerin değiştiğini ve değiştiğini gösterir.</span><span class="sxs-lookup"><span data-stu-id="1553f-153">If a property value changes via some external force (in a way other than by calling methods on the object), raise events indicate to the developer that the value is changing and has changed.</span></span> <span data-ttu-id="1553f-154">`Text`Bir metin kutusu denetiminin özelliği iyi bir örnektir.</span><span class="sxs-lookup"><span data-stu-id="1553f-154">A good example is the `Text` property of a text box control.</span></span> <span data-ttu-id="1553f-155">Kullanıcı öğesine metin yazdığında `TextBox` , özellik değeri otomatik olarak değişir.</span><span class="sxs-lookup"><span data-stu-id="1553f-155">When the user types text in a `TextBox`, the property value automatically changes.</span></span>
