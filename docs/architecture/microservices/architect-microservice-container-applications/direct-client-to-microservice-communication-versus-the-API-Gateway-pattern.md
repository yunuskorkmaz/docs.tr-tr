<span data-ttu-id="eed02-236">API Management bir sistem tarafından sunulan Öngörüler, API 'lerinizin nasıl kullanıldığını ve nasıl çalıştığını anlayabilmeniz için size yardımcı olur.</span><span class="sxs-lookup"><span data-stu-id="eed02-236">The insights available from an API Management system help you get an understanding of how your APIs are being used and how they are performing.</span></span> <span data-ttu-id="eed02-237">Bu, neredeyse gerçek zamanlı analiz raporlarını görüntülemenize ve işinizi etkileyebilecek eğilimleri tanımlamaya izin vererek bunu yapabilirler.</span><span class="sxs-lookup"><span data-stu-id="eed02-237">They do this by letting you view near real-time analytics reports and identifying trends that might impact your business.</span></span> <span data-ttu-id="eed02-238">Ayrıca, daha fazla çevrimiçi ve çevrimdışı analiz için istek ve yanıt etkinliği ile ilgili günlüklere sahip olabilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="eed02-238">Plus, you can have logs about request and response activity for further online and offline analysis.</span></span>

<span data-ttu-id="eed02-239">Azure API Management ile API 'lerinizi bir anahtar, belirteç ve IP filtrelemesi kullanarak güvenli hale getirebilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="eed02-239">With Azure API Management, you can secure your APIs using a key, a token, and IP filtering.</span></span> <span data-ttu-id="eed02-240">Bu özellikler esnek ve ayrıntılı kotaları ve oran sınırlarını zorlamanıza, ilkeleri kullanarak API 'lerinizin şeklini ve davranışını değiştirmenize ve yanıt önbelleğe alma ile performansı iyileştirmenize olanak tanır.</span><span class="sxs-lookup"><span data-stu-id="eed02-240">These features let you enforce flexible and fine-grained quotas and rate limits, modify the shape and behavior of your APIs using policies, and improve performance with response caching.</span></span>

## <a name="drawbacks-of-the-api-gateway-pattern"></a><span data-ttu-id="eed02-251">API ağ geçidi deseninin dezavantajları</span><span class="sxs-lookup"><span data-stu-id="eed02-251">Drawbacks of the API Gateway pattern</span></span>

- <span data-ttu-id="eed02-252">En önemli sakıncası, bir API ağ geçidini uyguladığınızda bu katmana iç mikro hizmetlerle geçidinizin verilir.</span><span class="sxs-lookup"><span data-stu-id="eed02-252">The most important drawback is that when you implement an API Gateway, you're coupling that tier with the internal microservices.</span></span> <span data-ttu-id="eed02-253">Bunun gibi bir başvuru, uygulamanız için ciddi zorluklar ortaya çıkarabilir.</span><span class="sxs-lookup"><span data-stu-id="eed02-253">Coupling like this might introduce serious difficulties for your application.</span></span> <span data-ttu-id="eed02-254">Azure Service Bus ekibinden bir mimari olan Clemens vaster, "[mesajlaşma ve mikro hizmetler](https://www.youtube.com/watch?v=rXi5CLjIQ9k)" oturumunda "yeni ESB" olarak, goto 2016 adresindeki bu olası zorluklar anlamına gelir.</span><span class="sxs-lookup"><span data-stu-id="eed02-254">Clemens Vaster, architect at the Azure Service Bus team, refers to this potential difficulty as "the new ESB" in the "[Messaging and Microservices](https://www.youtube.com/watch?v=rXi5CLjIQ9k)" session at GOTO 2016.</span></span>

- <span data-ttu-id="eed02-255">Mikro hizmetler API ağ geçidi kullanmak olası bir tek hata noktası oluşturur.</span><span class="sxs-lookup"><span data-stu-id="eed02-255">Using a microservices API Gateway creates an additional possible single point of failure.</span></span>

- <span data-ttu-id="eed02-256">API ağ geçidi, ek ağ çağrısı nedeniyle daha fazla yanıt süresi ortaya çıkarabilir.</span><span class="sxs-lookup"><span data-stu-id="eed02-256">An API Gateway can introduce increased response time due to the additional network call.</span></span> <span data-ttu-id="eed02-257">Ancak, bu ek çağrı genellikle iç mikro hizmetleri doğrudan çağıran çok geveze bir istemci arabirimine sahip olmaktan daha az etkiye sahiptir.</span><span class="sxs-lookup"><span data-stu-id="eed02-257">However, this extra call usually has less impact than having a client interface that's too chatty directly calling the internal microservices.</span></span>

- <span data-ttu-id="eed02-258">Düzgün şekilde ölçeklenmiyorsa, API Gateway bir performans sorunu olabilir.</span><span class="sxs-lookup"><span data-stu-id="eed02-258">If not scaled out properly, the API Gateway can become a bottleneck.</span></span>

- <span data-ttu-id="eed02-259">API ağ geçidi, özel mantık ve veri toplamayı içeriyorsa ek geliştirme maliyeti ve gelecekteki bakım gerektirir.</span><span class="sxs-lookup"><span data-stu-id="eed02-259">An API Gateway requires additional development cost and future maintenance if it includes custom logic and data aggregation.</span></span> <span data-ttu-id="eed02-260">Geliştiriciler, her mikro hizmetin uç noktalarını göstermek için API ağ geçidini güncelleştirmelidir.</span><span class="sxs-lookup"><span data-stu-id="eed02-260">Developers must update the API Gateway in order to expose each microservice's endpoints.</span></span> <span data-ttu-id="eed02-261">Üstelik, iç mikro hizmetlerde yapılan uygulama değişiklikleri API ağ geçidi düzeyinde kod değişikliklerine neden olabilir.</span><span class="sxs-lookup"><span data-stu-id="eed02-261">Moreover, implementation changes in the internal microservices might cause code changes at the API Gateway level.</span></span> <span data-ttu-id="eed02-262">Ancak, API Gateway yalnızca güvenlik, günlüğe kaydetme ve sürüm oluşturma (Azure API Management kullanırken olduğu gibi) uygualıyorsa, bu ek geliştirme maliyeti uygulanmayabilir.</span><span class="sxs-lookup"><span data-stu-id="eed02-262">However, if the API Gateway is just applying security, logging, and versioning (as when using Azure API Management), this additional development cost might not apply.</span></span>

- <span data-ttu-id="eed02-263">API Gateway tek bir takım tarafından geliştirilirse, bir geliştirme sorunu olabilir.</span><span class="sxs-lookup"><span data-stu-id="eed02-263">If the API Gateway is developed by a single team, there can be a development bottleneck.</span></span> <span data-ttu-id="eed02-264">Bu, farklı istemci ihtiyaçlarına yanıt veren birkaç fined API ağ geçidine sahip olmak için daha iyi bir yaklaşıma neden olur.</span><span class="sxs-lookup"><span data-stu-id="eed02-264">This is another reason why a better approach is to have several fined-grained API Gateways that respond to different client needs.</span></span> <span data-ttu-id="eed02-265">Ayrıca, API ağ geçidini dahili mikro Hizmetlerde çalışan farklı takımların sahip olduğu birden çok alana veya katmana ayırabilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="eed02-265">You could also segregate the API Gateway internally into multiple areas or layers that are owned by the different teams working on the internal microservices.</span></span>
